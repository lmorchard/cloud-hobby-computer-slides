What is a hobby computer?

Around the time I was born, home computers were sold in kit form. You got circuit boards and a bag of parts. You assembled the thing yourself and learned a bunch about hardware and software. This picture is of the Rum 80, a computer built from scratch in the 80s by Lumir Vanek from the Czech Republic. That's even cooler, because this guy made up his own kit. image credit - http://www.valachnet.cz/lvanek/diy/old_story/index.html

What is free?

Lots of folks around here are familiar with free as in beer versus free as in speech. Well, the computers I'm talking about are the free beer. Your software is the free speech. I think it's crucial to consider whether any particular offering of free beer helps you in your freedom of expression, or whether it serves to lock you in. And of course, these computers are free with asterisks. But, I think you can and should navigate these asterisks for fun and learning. image credit - http://www.sfreporter.com/santafe/article-7361-free-beer.html

What is yours?

And that leads me to the issue of what is yours. I've heard it said that if you can't fix it, you don't own it.
I would also say that it's only yours if you can pick it up and move it somewhere else. At least, in the case of software. Houses might be a bit harder to move.

Borrowing Other People's Computers for Your Hobbies

by Les Orchard / @lmorchard

When you're using the cloud, that's what you're doing: You're borrowing other people's computers and you'll have to give them back at some point. Often that time is measured in milliseconds.
But, whenever you have your hands on a tool, borrowed or not, you can end up learning and making some useful things.
Really, what I'm going to attempt to do is describe an approach to thinking about and evaluating cloud services: If it's free, what's the catch? Will you learn something useful in using it? Can you walk away cleanly with stuff you've made and put it somewhere else? Or, are you locked in and stuck when you have to give back what you've borrowed?
image credit - http://yoo-hoo-queen.blogspot.com/2011/05/brother-bugs.html

GitHub

One of the first things I want to do with the cloud is find a place to put my stuff.
And, since most of my hobbies tend to result in software of some kind, GitHub comes in very handy.
It's kind of like the Facebook of code, with activity streams and comments and such.

GitHub + Raspberry PI = <3

For example, when I was having fun hacking around with my Raspberry PI, I stashed all my code and experiments on GitHub.
This turned out really nice, because it meant I could edit things easily on my laptop
And when I sometimes wiped the SD card for the PI, it was really easy to just pull down a fresh clone of my code to get back to playing
And since all of this is on the web, I could point friends at some some of the code to ask for advice or just show off

Travis CI

So, the intent of Travis CI is run your tests for your code.
What you can see in this screenshot is Travis starting up a Linux machine and running the tests for a video game I was working on.
The neat thing is that it integrates with GitHub. Every time I push out a change, Travis gets a notification via the GitHub API and schedules a fresh test run.
Travis will also get notifications when someone has submitted a Pull Request to propose a change to my project, and so kick off tests to tell me whether this new contribution passes the tests.
image credit - http://blog.travis-ci.com/2013-07-17-introducing-travis-our-new-mascot/

Travis CI + GitHub Pages = <3

But, there are other things Travis can do before, during, or after running tests.
Something I wanted to do with this video game is to publish a new version of it to GitHub Pages, every time I pushed out a change that passed all my tests.
So, here, you can see some complicated glue code I wrote which does just that. The neat thing is that Travis let's you put encrypted things into the configuration so that you can put things like a GitHub API key in there to give Travis access to push to GitHub Pages on your behalf.

Travis CI

  • Free as in beer
  • Works with GitHub
  • Runs your code & tests
  • Slow during peak usage hours
free - except for private repos
github - reacts to changes in your repos
runs your code - you don't have to do much of anything special. the same code and tests that would run on your computer at home will run on travis.
slow - it will sometimes take awhile for travis to get around to running your tests during peak usage hours. but, you can't really complain too much about free computing power

Heroku

Heroku is a nifty platform for hosting web applications. I use it a lot for little node.js projects and experiments.
image credit - https://www.heroku.com/home

Heroku + GitHub = <3

And one of the coolest recent features on Heroku is this purple button.
You put it in the README of a GitHub repo. And, when you click it, Heroku automagically grabs your project and deploys it on their servers.

Heroku + Travis CI = <3

And like that thing I did earlier with GitHub Pages, you can also configure Travis to deploy a new version of your app to Heroku whenever tests pass.

Heroku

  • Free as in beer
  • Works with GitHub
  • Works with Travis CI
  • Your code works elsewhere
  • Apps shut down after inactivity
free beer - for very small apps, heroku is free. you can have a little bit of database space, and one web server working for you
your code - Heroku makes a few more demands on your code than Travis, but they're not very big demands. For the most part, whatever you have working on Heroku can be moved to another service provider with just a little bit of tweaking.
inactivity - if you don't use an app for awhile, Heroku will shut it down. It will start up again the next time you try to use it, but there will be a long pause as things come back awake.

IFTTT

* ifttt is handy and I use it for automating some social media stuff * But, it's a bad example of the kind of cloud service I really want to be using * While it *is* free as in beer, there is no direct open source equivalent to it and all of the triggers and rules I've set up on it are stuck there. If IFTTT ever goes away, so does all my stuff. image credit - http://lhshelpdesk.com/2015/01/29/ifttt-by-matt-germano/

Google Docs

* I use Google docs spreadsheets to help me make internet spaceship money in EVE Online * While there are lots of spreadsheets out there, some even Open Source, my particular usage here is fairly tied to Google Docs. It uses Google's own proprietary formula functions to access web APIs to look up market prices in game. * If Google Spreadsheets ever goes away, I will probably use any work I've done here. So, I try not to do too much here beyond kind of throwaway scratchpad work. Anything more than that, and I try to shift the effort into writing my own tools that work on computers I own, or on Heroku. image credit - http://googledocs.blogspot.com/

Amazon Web Services

* Amazon Web Services is kind of like the uber cloud. Almost any kind of computer you want to borrow, they have it in Virtual Machine form. * For example, I wanted to try running an old-school DOS BBS but make it accessible from the internet instead of from a dial-up phone line. But, it only worked on Windows 2000 or earlier. Well, it turns out they have ready made computers with Windows 2000 installed. So, I fired one of those up and spent a weekend playing with BBS software. * AWS has a free tier that you can use for a year as a new customer. Me, I've been a customer since early 2006, before there even was a free tier. But, you can do quite a lot with that before your first year runs out. * Still, that said, much of AWS is so cheap for personal use that it's almost-free beer. * For example, I moved my personal blog.lmorchard.com from WordPress to my own system that publishes to Amazon S3. I pay a few pennies a month to keep it running.
